# BDD: Módulo Identity
**Data:** 2026-06-17
**Status:** MVP Robust / Production-Ready
**Referência:** Domain Modeling Made Functional (DMMF)

## Objetivo
Gerenciar identidades de usuários, autenticação, perfis e vinculação de canais de comunicação (Telegram/WhatsApp).

## Fluxo 1: Estabelecimento de Principal
**Funcionalidade:** Identificação segura do usuário nas requisições.

**Cenário:** Resolução de identidade por ID de canal
- **Dado** que uma mensagem chega do Telegram com ID `123456`
- **Quando** o sistema tenta resolver o Principal
- **Então** o `UserID` associado a esse canal deve ser identificado
- **E** o contexto de segurança deve ser populado para o restante do workflow.

## Fluxo 2: Vinculação de Canais (WhatsApp)
**Funcionalidade:** Expansão de canais de interação.

**Cenário:** Novo número de WhatsApp vinculado ao usuário
- **Dado** um usuário autenticado
- **Quando** ele solicita a vinculação de um novo número de WhatsApp
- **Então** o registro deve ser salvo na tabela `identity.whatsapp_history_entries`
- **E** o canal deve ser marcado como preferencial, se solicitado.

## Fluxo 3: Exclusão Lógica (Soft Delete)
**Funcionalidade:** Privacidade e conformidade (LGPD).

**Cenário:** Usuário solicita encerramento de conta
- **Dado** um usuário ativo
- **Quando** o comando de exclusão é executado
- **Então** o usuário deve ser marcado como `Deleted`
- **E** os eventos de auditoria devem ser anonimizados conforme a política de privacidade.

## Regras de Domínio (DMMF)
- **Smart Constructors:** O `Email` e `WhatsAppNumber` devem ser validados na criação.
- **Value Objects:** O `UserID` é um tipo opaco que garante a não-troca de IDs entre domínios.
- **Pure Core:** A lógica de decisão de canal preferencial é baseada em regras puras sobre o histórico do usuário.

## Validação de Produção
- [ ] Verificar performance da resolução de Principal em alta carga.
- [ ] Garantir que dados sensíveis sejam mascarados nos logs de auditoria.
- [ ] Validar que usuários deletados não consigam realizar novas operações.
