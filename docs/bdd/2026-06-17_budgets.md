# BDD: Módulo Budgets
**Data:** 2026-06-17
**Status:** MVP Robust / Production-Ready
**Referência:** Domain Modeling Made Functional (DMMF)

## Objetivo
Gerenciar limites de gastos mensais, alertas de threshold e ingestão de despesas externas, garantindo que o usuário tenha visibilidade em tempo real do seu orçamento.

## Fluxo 1: Criação e Ativação de Orçamento
**Funcionalidade:** Definição de metas de gastos por categoria.

**Cenário:** Novo orçamento criado como rascunho e ativado
- **Dado** que um usuário define um limite de R$ 1000,00 para a categoria 'Alimentação'
- **Quando** o orçamento é ativado
- **Então** o registro deve persistir com status `Active`
- **E** um evento `BudgetActivated` deve ser disparado para o módulo de notificações.

## Fluxo 2: Alertas de Limite (Threshold)
**Funcionalidade:** Notificação proativa de gastos excessivos.

**Cenário:** Gasto atinge 80% do orçamento
- **Dado** um orçamento de R$ 1000,00
- **E** uma nova despesa de R$ 850,00 é registrada
- **Quando** o sistema avalia os alertas
- **Então** um estado de alerta `Warning` deve ser criado para o orçamento
- **E** uma notificação deve ser enviada ao usuário.

## Fluxo 3: Ingestão de Despesas Externas
**Funcionalidade:** Sincronização automática de gastos.

**Cenário:** Despesa vinda de integração externa (ex: Banco)
- **Dado** que uma despesa externa é recebida
- **Quando** o sistema processa a ingestão
- **Então** a despesa deve ser vinculada ao orçamento correspondente
- **E** o saldo disponível deve ser recomputado imediatamente no banco de dados.

## Regras de Domínio (DMMF)
- **Estado como Tipo:** Orçamentos podem ser `Draft`, `Active` ou `Archived`. Operações de despesa só são permitidas em `Active`.
- **Pure Core:** O cálculo de saldo restante é uma função pura que não depende de IO.
- **Workflows:** O processo de avaliação de alertas é um pipeline: `LoadBudget -> CheckThresholds -> CreateAlerts -> Notify`.

## Validação de Produção
- [ ] Validar precisão decimal nos cálculos de saldo (evitar erros de float).
- [ ] Verificar se eventos pendentes (PendingEvents) são processados de forma idempotente.
- [ ] Garantir que o reaper de eventos limpe dados antigos conforme a política de retenção.
