## Subtarefas

### 8.1 — Allowlist de IPs Meta no rate-limit (RF-21)

**Caddy (deployment/caddy/Caddyfile + Caddyfile.ratelimit):**
- Adicionado snippet `(backend)` para evitar duplicação do `reverse_proxy`
- Adicionado matcher `@meta_webhook` (`path /api/v1/whatsapp/*` + `remote_ip` nos 14 CIDRs Meta)
- Requests de IPs Meta para o webhook vão via `handle @meta_webhook { import backend }` — sem rate-limit
- Requests de outros IPs passam pelo `handle { import rate-limit; import backend }` — proteção preservada
- Mesma estrutura em ambos os Caddyfiles (padrão e ratelimit)

**Go middleware (internal/onboarding/.../middleware/rate_limit.go):**
- Adicionado campo `allowlist []*net.IPNet` ao `RateLimiter`
- Assinatura `NewRateLimiter` extendida com `allowlistCIDRs ...[]string` (variadic, retrocompatível)
- `isAllowlisted(ip string) bool` verifica se o IP está em algum CIDR da allowlist
- `Middleware` bypassa rate-limit para IPs allowlistados antes de verificar bucket

**Config (configs/config.go):**
- Campo `WebhookMetaAllowlistCIDRs string` adicionado a `WhatsAppConfig`
- Env key `WHATSAPP_WEBHOOK_META_ALLOWLIST_CIDRS` registrado em `envKeys()`
- Default: 14 CIDRs oficiais da Meta (fonte: https://developers.facebook.com/docs/graph-api/webhooks/ip-addresses)

**Wiring (cmd/server/whatsapp_wiring.go):**
- `NewRateLimiter` recebe `stringsutil.ParseCSV(cfg.WhatsAppConfig.WebhookMetaAllowlistCIDRs)` como 4º argumento

**Testes (rate_limit_test.go):**
- Adicionado `TestAllowlist` com 3 cenários:
  - IP em allowlist passa mesmo após esgotar bucket (20 requests, todos 200)
  - IP fora da allowlist é limitado normalmente (429 no 11º request)
  - Sem allowlist configurada: comportamento original preservado

### 8.2 — pg-tunnel bind loopback (RF-22)

**deployment/compose/compose.swarm.yml:**
```yaml

    published: "127.0.0.1:15432"  # era: 15432 (0.0.0.0)
```

O pg-tunnel não é mais acessível de fora do host. Acesso DBA requer SSH tunnel.
