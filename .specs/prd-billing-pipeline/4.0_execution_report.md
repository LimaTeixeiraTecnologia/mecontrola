## Decisões Técnicas

1. `BindUser(ctx, subID, userID)` adicionado à interface `SubscriptionRepository` — o índice único parcial `billing_subscriptions_user_active_uniq_idx` só dispara em writes com `user_id NOT NULL`. O método `UpsertByOrder` sempre passa `user_id=nil` (E3 não existe no MVP). `BindUser` é o ponto natural onde RF-17 dispara.

2. `Hydrate()` adicionado a `entities.Subscription` — a entidade original não tinha `id`. O repositório precisa retornar o `id` do banco para que `ExtendPeriod` e `ApplyTransition` funcionem com o caller real.

3. `ParseStatus()` adicionado a `valueobjects/status.go` — necessário para reconstruir entidade a partir do banco.

4. `MarkSuperseded` adicionado a `ProcessedEventRepository` — previsto na task 4.2 (RF-12) mas ausente da interface 3.0. Decisão: estender a interface é o caminho correto; alternativa seria SQL direto no use case (proibido por arquitetura).

5. `MarkProcessed` adicionado a `KiwifyEventRepository` — complementa o `Persist` para fechar o ciclo de auditoria.

6. `resolvePlanFromCode` como função de pacote (não método) — sem estado, usada apenas no scan, não precisa de receiver.

## Riscos Residuais

- `resolvePlanFromCode` usa durations hard-coded (30/90/365) como fallback para o scan. Em teoria, se os planos do banco tiverem durações diferentes dos seeds, haverá divergência. Porém, os planos são seeded na migration 0004 com esses valores e são imutáveis no MVP. O `FindByCode` e `FindByKiwifyProductID` no plan_repository retornam a duração real do banco.
- O scan em `subscription_repository` usa `resolvePlanFromCode` (durações fixas) em vez do `plan_repository`. Isso é aceitável no MVP pois os planos são imutáveis. Seria resolvível passando o `plan_repository` ao `subscription_repository` se durações variáveis fossem necessárias.
