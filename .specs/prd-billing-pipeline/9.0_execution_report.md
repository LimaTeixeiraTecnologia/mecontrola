## Decisões Técnicas

1. **Checkpoint no job vs no use case**: O `ReconciliationJob.Run` lê o checkpoint via `factory.ReconciliationCheckpointRepository(db)` antes de invocar o use case; o checkpoint é avançado dentro do use case (`ReconcileSubscriptions.Execute`) apenas após todas as páginas processadas sem erro fatal — conforme especificado.

2. **Metric `billing_reconciliation_corrections_total`**: Registrada no construtor do `ReconciliationJob` via `o11y.Metrics().Counter(...)`. O job delega toda lógica de reconciliação ao use case existente; a métrica acompanha erros fatais de run.

3. **`DeleteOlderThan` com subquery LIMIT**: A implementação usa `DELETE ... WHERE envelope_id IN (SELECT ... LIMIT N)` para batching eficiente sem lock de tabela inteira, padrão seguido pelo outbox housekeeping.

4. **Context cancellation**: O `KiwifyEventsHousekeepingJob` verifica `ctx.Done()` no início de cada iteração, garantindo shutdown cooperativo.

5. **Campos de BillingConfig**: Adicionados `KiwifyEventsRetentionDays` (default 90), `KiwifyEventsHousekeepingSchedule` (default `@daily`), `KiwifyEventsHousekeepingBatch` (default 500) — sem alterar a lógica de validação existente (zero = não configurado).

## Riscos Residuais

- A checagem `ctx.Done()` no housekeeping ocorre ANTES de cada `DeleteOlderThan`, não durante. Se `DeleteOlderThan` demorar muito em uma batch grande, o shutdown cooperativo aguarda o fim da batch atual.
- Integration tests usam `kiwify_sales` como nome de checkpoint fixo; testes paralelos podem conflitar se rodarem no mesmo banco simultaneamente (mitigado por uso de `UnixNano` nos IDs dos dados de teste).
