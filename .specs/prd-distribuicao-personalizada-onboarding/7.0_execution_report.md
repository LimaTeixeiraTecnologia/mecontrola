## Bug encontrado e corrigido

Durante a execução do gate de integração completo (`go test -tags integration ./internal/agents/...`), o teste `TestWhatsAppInboundConsumerIntegrationSuite/TestInteg_OnboardingFluxoDeCartao_CriaUmUnicoCartaoSemLoop` falhou. Causa raiz: o handler `handleReviewAwaitDistribution` (implementado nas tarefas 3.0/4.0, `onboarding_workflow.go:1349-1392`) agora chama `classifyDistributionIntent` (ADR-002, pré-classificador de intenção) **antes** de decidir se extrai valores — uma chamada `a.Execute` adicional em relação ao baseline anterior à entrega. O teste de integração `whatsapp_inbound_consumer_integration_test.go` mockava a sequência de `a.On("Execute", ...).Once()` com o schema antigo (`action: "confirm"`, campos `custo_fixo` etc. de `extractAllocationValues`), que nunca foi migrada para o novo schema `distributionIntentExtract` (`action: "accept"|"personalize"|"values"`, `mixed_unit`). Como `"confirm"` não é um valor válido de `distributionIntentKind` (`ParseDistributionIntentKind`, `onboarding_workflow.go:211-222`), o handler caía no branch `default` ("não entendi sua resposta"), quebrando a sequência de turnos do teste.

Correção aplicada (arquivo `internal/agents/infrastructure/messaging/database/consumers/whatsapp_inbound_consumer_integration_test.go`): substituído o mock `allocationExtract` (`action: "confirm"`) por `distributionIntentAcceptExtract` (`action: "accept", "mixed_unit": false`), consistente com o contrato real de `classifyDistributionIntent`. Diff mínimo e cirúrgico — apenas dados de teste, nenhuma alteração de código de produção. Após a correção, `go test -tags integration ./internal/agents/... -count=1` passa integralmente. Este achado está dentro do escopo da tarefa 7.0 (validação de não-regressão): o teste estava desatualizado em relação ao comportamento novo introduzido pela tarefa 3.0/4.0, e a tarefa 7.0 existe justamente para capturar esse tipo de lacuna antes do merge.
