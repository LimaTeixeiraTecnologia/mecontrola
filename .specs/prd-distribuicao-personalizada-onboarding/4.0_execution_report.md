## Suposições
- A mensagem de `mixed_unit` (`distributionMixedUnitPrompt`) não ecoa os valores por categoria, pois o `distributionIntentSchema` (schema fechado apenas com `action`+`mixed_unit`, ADR-002) não extrai valores — a extração de valores só ocorre depois, no caminho `values`, e o design de ADR-002 explicitamente curto-circuita o `mixed_unit` **antes** da extração para não introduzir uma terceira chamada LLM nem duplicar o schema compartilhado. RF-10/RN-13 mencionam "ecoa o que entendeu", mas isso é uma tensão já aceita e documentada no ADR-002 (trade-off de zero-regressão vs. eco completo); registrado aqui para visibilidade, não corrigido por estar fora do contrato assinado pela ADR já aprovada.
- `handleReviewAwaitPersonalize` reaproveita `activateAllocationValues` (a mesma função usada pelo caminho `values` de `handleReviewAwaitDistribution`) para evitar duplicar a lógica de decisão de saldo/conversão/persistência entre os dois handlers; quando `activateAllocationValues` decide re-suspender por over/under, ela sempre volta para `reviewAwaitDistribution` (comportamento correto para o caminho normal) — por isso `handleReviewAwaitPersonalize` remapeia esse retorno para `reviewAwaitPersonalize` quando aplicável (linhas 1361-1364), preservando a subtarefa 4.2 ("over/under permanecem no sub-estado" refere-se ao sub-estado atual do usuário, que é personalizar).
- A precedência de recusa/ambíguo em `handleReviewAwaitPersonalize` foi implementada verificando `kind == allocationInputConfirm && soma <= 0` (mesma heurística usada em `DecideAllocationKind` para colapsar para "confirm" quando não há valores usáveis) — isso cobre tanto uma recusa explícita quanto uma resposta que o LLM classificou como "confirm" sem números, repetindo a orientação de personalizar (`personalizeReprompt`) em vez de aplicar a distribuição padrão (que seria o comportamento de `accept`, não aplicável dentro do modo personalizar).
- Optei por não introduzir o parâmetro `dist observability.Counter` nos handlers nesta tarefa — a técspec o menciona na assinatura alvo, mas a subtarefa 4.3 desta task file diz explicitamente "mantendo `reviewAwaitDistribution`/`reviewAwaitConfirm` inalterados" e a introdução do contador é escopo explícito da tarefa 5.0 (tasks.md: "5.0 depende de 4.0"). Introduzir o parâmetro agora sem uso real geraria wiring incompleto e churn duplo em `module.go`/`BuildOnboardingWorkflow` nesta e na próxima tarefa.

## Riscos Residuais
- **Achado de revisão `[medium]`, corrigido nesta execução**: a primeira versão de `activateAllocationValues` usava `renderBalanceMessage(balance, values)` como fallback genérico também para o caso em que `DecideAllocationsBP` falha por um motivo **estrutural** (ex.: categoria negativa) mesmo com `balance.Status == distributionBalanced` (soma bruta correta, mas uma categoria individual negativa). Isso produziria a mensagem enganosa "A distribuição precisa fechar 100% no total" em vez de comunicar o problema real (valor negativo). Corrigido distinguindo `errors.Is(decErr, errAllocationOutOfTolerance)` (usa `renderBalanceMessage`) de outros erros estruturais (usa o texto direto do `decErr`, preservando o comportamento pré-refactor de mensagens específicas). Coberto pelo novo cenário "resume em reviewAwaitPersonalize com categoria negativa dentro da tolerancia deve reprompt com o motivo estrutural, mantendo mesmo sub-estado" (`PASS`).

Se o produto exigir eco literal dos valores no aviso de `mixed_unit`, isso exigiria uma terceira chamada LLM ou estender `distributionIntentSchema`, fora do escopo/ADR desta tarefa; deixado como decisão de produto para revisão futura, não como defeito.
