## Suposições
- O acoplamento das tools a `interfaces.CardManager` (já usado por `resolve_card`/`list_cards`/`get_card`/`update_card`/`delete_entry`) é o ponto de leitura correto e já disponível em `buildFinancialTools` (`cards interfaces.CardManager`), evitando qualquer chamada SQL nova na tool (R-AGENT-WF-001.2).
- O guard `card_provenance` inspeciona apenas nomes/ordem de `ToolCallRecord`, sem depender do payload (`found=false` dentro do JSON de `resolve_card` não é lido pelo guard); a rejeição por card inexistente é responsabilidade exclusiva da camada de validação de existência na tool, conforme decisão do ADR-003 (duas camadas complementares e independentes).
- Para os testes de integração/e2e que não setavam `CardID`, um `CardManager` stub/mocado sem expectativas de `GetCard` é suficiente, pois o código só invoca `cards.GetCard` quando `in.CardID != ""`.
