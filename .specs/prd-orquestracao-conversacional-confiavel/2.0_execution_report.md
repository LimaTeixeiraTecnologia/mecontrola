## Suposições
- RF-28 ("run sem scorer executado") foi interpretado como responsabilidade de `internal/agents/application/agents/scoring_hooks.go` (consumidor), não do runtime genérico `internal/platform/agent/runtime.go`, pois o runtime não tem conhecimento semântico de "scorer" — apenas invoca `Hooks.AfterExecute` de forma opaca. Mudança tratada como aditiva e mínima (não altera captura de `args`, que é escopo da tarefa 5.0/ADR-004).
- O "fallback seguro" textual (RF-22/2.6) já existia no consumidor (`whatsapp_inbound_consumer.go`, `fallbackReply = "não consegui concluir agora, pode repetir?"`) e se aplica automaticamente a qualquer `!outcome.Succeeded()`, incluindo o novo caso de truncamento — não foi necessário introduzir novo texto fixo no runtime, que já retorna `Content=""` em todo caminho de falha.
- `AGENT_MECONTROLA_MAX_TOKENS` foi adicionado como campo próprio de `AgentConfig` e `Deps.AgentMaxTokens` (distinto de `AgentConfig.MaxTokens`/`AGENT_LLM_MAX_TOKENS`, que é o teto de request do provider LLM, não o `WithDefaultMaxTokens` do agente) para não conflatar dois conceitos com semânticas diferentes já existentes no código.
- Limite de validação em produção para `AGENT_MECONTROLA_MAX_TOKENS` foi fixado em `(0..8192]`, seguindo o padrão já usado para `AGENT_LLM_MAX_TOKENS` (`(0..4096]`), com teto maior por ser aproximadamente 2x o valor de referência.
