## Suposições
- **Comparação "por versão do agente"** (techspec, seção "Monitoramento e Observabilidade"): o runtime atual não expõe um label de versão de agente distinto de `agent_id` (que é a constante fixa `mecontrola-agent`, não um discriminador semântico de versão/deploy). A comparação por versão foi implementada via **janela temporal** (`since` = timestamp do deploy, filtrando `platform_runs.started_at >= since`) combinada com **anotação de deploy** no dashboard Grafana — abordagem consistente com o padrão "gate pós-deploy monitora agregados produtivos" do ADR-005, que não exige um label de versão dedicado. Introduzir um label de versão exigiria mudança de schema/runtime (fora de escopo desta tarefa, que é "consulta/observabilidade, não código de runtime" conforme a própria task file). Documentado explicitamente no runbook (seção 1) e no dashboard (descrição).
- **`RunUpdateErrors`/`MessageAppendErrors` via `PrometheusCounters` manual, não `AggregateReader`**: esses dois contadores existem apenas como métrica Prometheus (`internal/platform/agent/runtime.go`), sem coluna dedicada em `platform_runs` que discrimine "falha de update" como causa (o run em si é marcado `failed`/`succeeded`, mas o motivo específico do erro de update não é persistido em coluna própria). Por isso `ComputeGate` recebe esses dois valores como parâmetro (`PrometheusCounters`), documentado como limitação conhecida na seção 7 do runbook, com a consulta PromQL equivalente fornecida para leitura manual ou integração futura com a API do Prometheus.
- **`R-DTO-VALIDATE-001` não aplicável**: os DTOs internos deste pacote (`RunAggregate`, `ScorerAggregate`, `OperationalCounters`) não são input DTOs de fronteira HTTP/tool — são estruturas de leitura/cálculo internas ao gate de observabilidade, isentas pela cláusula "DTOs construídos exclusivamente internamente pelo use case" da regra.
- **Escopo restrito a consulta/observabilidade, sem wiring em `module.go`**: conforme a task file ("Gate pós-deploy é consulta/observabilidade + runbook, não código de runtime"), o pacote `postdeploy` não foi conectado ao wiring de produção (`internal/agents/module.go`) nem expõe endpoint HTTP/CLI — é consumido manualmentde por operador via runbook ou por chamada de código ad-hoc (seção 5.2 do runbook), preservando o contrato público existente (RF-55) sem adicionar superfície nova de API.

## Riscos Residuais
- As consultas de agregação (`aggregate_reader.go`) filtram por `agent_id` e `started_at >= $2` sem um índice composto dedicado `(agent_id, started_at)` em `platform_runs` — o índice existente é `(resource_id, started_at DESC)` e `(platform_thread_id, started_at DESC)`. Em volume alto (N≥100 runs é pequeno, mas o histórico total da tabela pode crescer), a consulta pode degradar para sequential scan filtrado. Fora de escopo desta tarefa alterar schema Postgres (PRD/techspec declaram explicitamente "sem alteração de schema Postgres — fora de escopo"); mitigação natural é a baixa volumetria esperada (janela de 14 dias, não histórico total) e o uso pontual (operador, não hot path). Registrar como candidato a ADR de índice se o volume de runs crescer substancialmente.
- O dashboard usa `agent_id` como variável de template para permitir filtragem futura, mas hoje há apenas um valor possível (`mecontrola-agent`) — a variável está pronta para o dia em que houver múltiplos agentes/versões nomeadas distintamente, sem necessidade de retrabalho do dashboard.
- O `no_duplicate_write` como sinal de "escrita duplicada" (`DuplicateWriteViolations`) depende do scorer comportamental homônimo (tarefa 5.0) estar corretamente configurado com `AlwaysSample()` em produção — se a amostragem for reduzida no futuro, o gate pode subestimar violações reais. Nenhuma ação necessária agora (amostragem atual é `AlwaysSample()`), mas documentado como dependência implícita.
- O runbook assume que o operador tem acesso direto ao Postgres de produção (via `psql` ou client SQL) para as consultas manuais da seção 5.1; não há CLI/comando automatizado dedicado nesta entrega (fora de escopo — "consulta/observabilidade, não código de runtime").
