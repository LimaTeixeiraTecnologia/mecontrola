## Riscos Residuais / Decisões Registradas
1. **RF-26 (resumo geral) implementado via `category_detail` sem `category`, não como tool dedicada.** A techspec lista apenas 5 tools novas em "Novas ferramentas" (`edit_budget_total`, `category_detail`, `cancel_plan_info`, `support_info`, `edit_goal`); RF-26 é mapeada a "`query_plan`/`GetMonthlySummary` + bloco geral" sem nomear artefato novo. Decisão: reaproveitar `category_detail` (campo `category` opcional) para cobrir RF-25 e RF-26 com a mesma tool, evitando inventar uma 6ª tool não listada em 8.1. Scorers/golden de RF-26 (task 9.0) devem validar essa escolha.
2. **`category_detail` cardinalidade de matching de categoria é heurística (nome/slug case-insensitive, exato então substring)** — não usa NLU/LLM; se o usuário citar um termo ambíguo, retorna `outcome=clarify`. Sem golden real-LLM disponível nesta tarefa para validar taxa de acerto (fica para o gate da task 9.0).
3. **Extensão de `MonthlyEntry` com categoria (transactions + agents)** foi tratada como mudança aditiva de baixo risco (mesma tabela, colunas já existentes, sem migration) para fechar o gap de dados de RF-25. Não estava listada explicitamente no "Domínio aditivo" da task 1.0; registrado aqui para auditoria — recomenda-se ADR retroativo se o cutover (9.0) exigir rastreabilidade formal.
4. **`destructive-manage` ganhou 2 operações (`DeleteEntry`, `UpdateRecurrence`) fora do escopo original da task 6.0** (que só cobria `DeleteCard`/`DeleteRecurrence`, RF-21). Necessário para cumprir literalmente a tabela "Modificar (não deletar)" da techspec, que lista `delete_entry.go`/`update_recurrence.go` apontando para `destructive-confirm`. Testes cobrindo os dois novos casos foram adicionados.
5. **Cenários de scoring (`SummaryScenario`/`GeneralScenario`, limiar 90% para "near_limit"/"attention")** em `category_scenario.go` são uma decisão de engenharia não especificada numericamente pelo PRD/techspec (que só descreve os 4 cenários qualitativamente). Documentado para revisão do time de produto/scorers na task 9.0.
6. **Lint pré-existente não corrigido**: `ParsePaymentMethod`/`PaymentMethod.String()` (task 1.0, `internal/transactions/domain/valueobjects/payment_method.go`) excedem complexidade ciclomática 15; arquivo não tocado por esta tarefa, fora do escopo de 8.0.
7. **Cutover/remoção física do legado é explicitamente task 9.0** (ADR-005). Arquivos `pending_entry_*.go`, `confirm_state.go`, `destructive_confirm_workflow.go`, `card_create_confirm_*.go`, `budget_creation_*.go`, `register_attempt.go` (usecase legado), `*_continuer.go` continuam no repositório, apenas fora do wiring ativo de `module.go` — compilam e têm testes próprios ainda passando, sem impacto em produção após deploy desta tarefa.
