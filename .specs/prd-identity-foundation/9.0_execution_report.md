## Suposições
- Docker daemon não está em execução neste ambiente de desenvolvimento; todos os outros testes de integração do projeto (outbox, database) apresentam o mesmo erro, confirmando que é pré-condição de infraestrutura e não defeito de código.
- O cenário TestMapperRejeitaUUIDCorrompido usa FindByWhatsAppNumber em vez de FindByID porque entities.NewUserID valida UUID v4 no lado da aplicação — não é possível construir um UserID v3 para passar ao método.
- O cenário TestDuplicateWhatsAppNumberConcorrente aceita dois resultados válidos: ambos retornam o mesmo ID (upsert idempotente venceu a race), ou um retorna ErrDuplicateWhatsAppNumber.
- fakeIDGenerator usa uuid.New() (geração aleatória de v4) para produzir IDs válidos e únicos nos testes.

## Riscos Residuais
- TestMapperRejeitaUUIDCorrompido usa FindByWhatsAppNumber para acionar o mapper com UUID v3; se o schema do banco mudar para rejeitar UUID não-v4 em nível de constraint, o teste precisará de outra estratégia de injeção.
- TestDuplicateWhatsAppNumberConcorrente com 2 goroutines pode ser flaky em runners com alto overhead de goroutines; se necessário, aumentar para 3-5 tentativas ou usar channel de sincronização explícita.
