## Suposições
- A VPS de produção/staging já terá Docker Swarm inicializado quando `deploy.sh`/`deploy-local.sh` forem executados; caso contrário, os scripts logam aviso e pulam a criação de secrets.
- As credenciais AWS (`AWS_ACCESS_KEY_ID`/`AWS_SECRET_ACCESS_KEY`) serão configuradas no `.env` da VPS para habilitar o backup S3; caso ausentes, o deploy loga aviso e continua.
- A aplicação Go continuará lendo os valores sensíveis via `.env` (`env_file`) nesta fase, conforme decisão híbrida do ADR-003.

## Riscos Residuais
- Rotação de secrets (`MODE=rotate`) atualiza os services em execução, mas o `compose.swarm.yml` referencia os secrets por nome estável (`mecontrola_<NOME>`). Após uma rotação, um novo `docker stack deploy` recriaria o service com o nome antigo; o operador deve atualizar o compose ou manter o secret antigo até a próxima revisão do arquivo.
- Os testes de upload S3 e de IAM restricto só podem ser validados em ambiente com bucket e credenciais reais (staging/produção).
