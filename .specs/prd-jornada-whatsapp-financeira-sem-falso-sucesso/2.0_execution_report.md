## Suposições
- `SeedExpireAfterFailedWrite` não incrementa `FailedWriteResumeCount` (expiração não consome orçamento de retry) — coerente com ADR-002.5 (expiração ≠ retry).
- Para `executeDirectWrite`, `DecidePostWrite(agent.ToolOutcomeRouted, ref.ID)` é semanticamente correto: caminho direto não tem conceito de replay, logo ID vazio ⇒ falha tipada; ID válido ⇒ Completed.
- `ShouldExpireAfterFailedWrite` restringe a `AwaitingSlotConfirmation` porque o único caminho que produz `RunStatusFailed` pós-escrita opera nesse slot; outros slots não alcançam run falho por escrita.
- Em run falho, o kernel (engine.finalizeFailed) devolve result.State = estado de ENTRADA do passo (confirm suspenso, ainda PendingStatusActive), preservando-o para o retry; por isso os testes de escrita vazia asseram PendingStatusActive + errors.Is(ErrWriteAcceptedWithoutResource) e NÃO o ResponseText de falha (a mensagem ao usuário é surfada na camada de inbound quando Continue retorna erro, coerente com o teste de falha permanente já existente na suíte).
