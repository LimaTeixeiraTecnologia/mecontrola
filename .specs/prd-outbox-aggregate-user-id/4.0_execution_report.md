## Suposições

- A billing `Subscription` entity não expunha `user_id` previamente. Optou-se por adicionar `HydrateWithUser()` (novo constructor) preservando `Hydrate()` por retrocompatibilidade com todos os callers de use cases que não têm acesso ao userID no momento de construção do objeto intermediário.
- Subscriptions recém-criadas (antes de `BindUser`) terão `AggregateUserID = ""` (NULL no DB), o que é aceito na v1 per ADR-001.
- A falha de lint pré-existente (`identity/domain/valueobjects/gateway_timestamp_test.go`) é arquivo não rastreado (untracked), fora do escopo desta tarefa e do nosso diff.

## Riscos Residuais

- Use cases que constroem `entities.Hydrate()` intermediários (ex: `renewed` em `process_subscription_renewed.go`) perdem o `userID` — eles passarão `AggregateUserID = ""`. Aceitável na v1 (ADR-001: NULL permitido). Resolução futura: migrar os callers para `HydrateWithUser()` carregando o userID do objeto `existing`.
- Billing subscriptions sem `user_id` (antes de `BindUser` no onboarding) publicarão eventos com `AggregateUserID` vazio. Comportamento esperado na v1.
