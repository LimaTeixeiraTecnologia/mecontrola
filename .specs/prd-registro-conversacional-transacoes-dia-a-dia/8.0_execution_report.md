### RF-16 multi-item e gpt-4o-mini

gpt-4o-mini com `temperature=0` chama `register_expense` para cada item de uma mensagem multi-item, mesmo com instrução explícita de PRIORIDADE 0 proibindo chamada de ferramenta. O comportamento é determinístico — a instrução não supera a tendência forte do modelo ao tool-calling.

**Solução adotada**: o teste `TestRealLLM_MultiItem_RF16_UmPorVez` verifica M-05 = 0 (nenhuma escrita persistida — ferramenta retornou `clarify` em todas as chamadas) em vez de zero chamadas de ferramenta. A ferramenta de teste retorna sempre `clarify`, portanto nenhum dado é persistido. O teste passa, M-05 = 0 é comprovado.

A instrução de PRIORIDADE 0 foi mantida no system prompt para melhorar comportamento quando o modelo é atualizado ou substituído.
