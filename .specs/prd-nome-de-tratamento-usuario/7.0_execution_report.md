## Gap de Produção Encontrado e Corrigido Durante a Validação

Ao rodar o gate real-LLM pela primeira vez (subtarefa 7.6), o caso "alterar nome de tratamento sem nome informado" falhou 3/3 vezes: o modelo (`openai/gpt-4o-mini`) respondia diretamente em texto livre ("Claro! Como você gostaria que eu te chamasse?") **sem chamar a tool** `edit_treatment_name`. Isso é um defeito real de produção: sem a chamada da tool, nenhum `Run`/workflow é iniciado, o estado de espera não é persistido, e a resposta do usuário à pergunta seguinte não seria resolvida pelo `ResumeDispatcher` (violaria R-AGENT-WF-001.7 — pending step obrigatório antes de pedir clarificação).

Diagnóstico isolado (script de debug local, removido antes de finalizar) confirmou que o texto do system prompt (`mecontrola_agent.go`) por si só não bastava; o fator decisivo foi a **descrição da própria tool** (visível ao LLM na definição da function-call). Correção aplicada em dois arquivos de produção reais (não apenas no harness de teste):
1. `internal/agents/application/tools/edit_treatment_name.go` — descrição da tool reforçada com "SEMPRE chame esta ferramenta... mesmo que o campo name venha vazio — NUNCA responda essa pergunta diretamente sem chamar a ferramenta".
2. `internal/agents/application/agents/mecontrola_agent.go` — instrução de roteamento reforçada com o mesmo imperativo e justificativa (persistência do estado de espera).

Após a correção, o gate real-LLM passou 9/9 (3 casos × 3 repetições) para `CategoryTreatmentName`, e a suíte completa (19 categorias + `tool_error`) permaneceu em 1.0000 em todas — confirmando ausência de regressão nas demais categorias causada pelo reforço textual no prompt compartilhado.

## Suposições
- O `Case` struct do pacote `golden` não possui campo `Metadata` (diferente de `scorer.RunSample.Metadata` usado pelos scorers em `internal/agents/application/scorers/`); a menção da techspec a `Metadata["requires_brand_emoji"]=true` foi interpretada como reuso da mesma lógica de verificação (asterisco simples + emoji oficial), implementada como uma `ResponsePropertyFunc` local (`toneAdherentConfirmation`), consistente com o padrão já estabelecido no arquivo (`nonEmptyResponse`, `containsAny`, `allOf`) — nenhum campo novo foi adicionado ao `Case` para não introduzir uma abstração paralela não solicitada pelo PRD/techspec.
- O stub de tool `edit_treatment_name` no harness real-LLM (`goldenEditTreatmentNameTool`) foi implementado como uma função dedicada (em vez do `goldenCaptureTool` genérico) porque o scorer de tom avalia a *resposta final do LLM*, e o LLM só consegue relayar corretamente o texto com negrito/emoji se a tool devolver essa mensagem realista — mesmo padrão já usado por `goldenRegisterIncomeConfirmTool`/`goldenCreateBudgetTool` no mesmo arquivo.
- A correção de produção (reforço de instrução na tool e no agente) foi limitada ao mínimo necessário para fazer o modelo chamar a tool de forma determinística mesmo sem `name`; não foi alterado nenhum outro comportamento da tool/produção além da string de descrição.
- Dado que nenhum caso golden pré-existente usa o catálogo completo de tools sem `ToolSubset` (confirmado via grep: 57 `ToolSubset:` para 58 `Name:` — a única exceção é pré-existente, fora do escopo desta tarefa), a adição do novo tool `edit_treatment_name` ao catálogo não tinha risco de vazamento para outras categorias; isso foi confirmado empiricamente pela execução completa do gate (19 categorias em 1.0000).
