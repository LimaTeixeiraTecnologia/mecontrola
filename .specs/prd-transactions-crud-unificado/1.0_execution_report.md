## Escopo Implementado

Camada de domínio pura da unificação: extensão do VO `PaymentMethod` com dois novos meios
(`vale_refeicao`, `vale_alimentacao`) e enriquecimento dos commands
`CreateTransaction`/`UpdateTransaction` com `CardID`, `Installments` e invariantes de crédito.
Nenhuma lógica de fatura/parcelamento/persistência — apenas tipos fechados e smart constructors
(ADR-001: diferenciação `credit_card` é orquestração a jusante, não branch aqui).

## Riscos Residuais / Suposições

- Sinalização de `Installments` ausente via `int == 0` → `None` (default 1 resolvido a jusante,
  Tarefa 4.0), conforme RF-14. Validação de superfície de `Raw*.Validate()` (card_id/installments na
  fronteira DTO) é deliverable de tarefa de DTO/input, não desta.
- Invariante `credit_card ⇒ outcome` só é avaliado quando o parse de `payment_method` e `direction`
  teve sucesso, evitando erro redundante quando o campo base já é inválido.
- Não há teste testify/suite whitebox: `internal/*/domain/` é explicitamente exceção de R-TESTING-001;
  seguido o padrão blackbox table-driven vigente nos ~20 arquivos de teste de domínio existentes.
