## Suposições
- `IsTransient(err)` classifica apenas erros de infraestrutura genéricos via stdlib (`context.DeadlineExceeded`, `sql.ErrConnDone`, `driver.ErrBadConn`, `net.Error.Timeout()`, `*net.OpError`) — deliberadamente **sem** importar `github.com/jackc/pgconn`/`pgerrcode` na camada `application`, pois nenhum outro arquivo `application/` do repositório importa o driver Postgres (esse acoplamento pertence a `infrastructure/postgres`); a lista é whitelist estrita conforme ADR-003 ("default é permanente"), coerente com o predicado testado (timeout/deadline/conn-reset ⇒ true; erro de validação/domínio ⇒ false).
- Erros de conexão específicos do pgx (`*pgconn.PgError` com código `08xxx`) não são classificados como transitórios por este predicado nesta tarefa; se necessário no futuro, a classificação por código SQLSTATE deveria entrar como novo predicado em `infrastructure/persistence` (camada correta para o tipo do driver) e ser injetada/composta aqui, não importada diretamente na `application`.
- Mantido o contrato de 3.0 (ADR-002): erro real de escrita — transitório exaurido ou permanente — sempre retorna `StepStatusFailed` + erro propagado (nunca re-suspende); ver Riscos Residuais para a tensão com o texto literal de ADR-003 §4 ("pending entry permanece retomável").
- Escopo do retry limitado estritamente a `executeWithIdempotency`/`executeDirectWrite` (subtarefa 4.2); `validateCategoryForWrite` (leitura/validação, já corrigida por 3.0) não recebeu retry — fora do escopo explícito da tarefa 4.0 e do ADR-003.

### Causa raiz confirmada

`Engine.Resume` (`internal/platform/workflow/engine.go:200-208`, inalterado) exige
`snap.Status == RunStatusSuspended`; `Store.Load` (`infrastructure/postgres/store.go`, inalterado)
filtra `status IN ('running','suspended')`. Um run `RunStatusFailed` (produzido pelo contrato
ADR-002/3.0 quando o retry de 4.0 esgota) é estruturalmente invisível a ambos — a próxima
confirmação do usuário caía em `no_run_found` silencioso em `PendingEntryContinuer.Continue`
(`result.Status == 0` → `Handled: false`), sem jamais reexecutar a escrita.

### Desenho da correção

1. **Kernel (`internal/platform/workflow`, genérico, sem semântica de domínio — R-WF-KERNEL-001
   preservado)**:
   - Novo método `Store.LoadLatest(ctx, workflow, key string) (Snapshot, bool, error)` — carrega o
     snapshot mais recente por `(workflow, correlation_key)` **independente de status**
     (`ORDER BY created_at DESC, id DESC`, sem tiebreaker ambíguo). Implementado em
     `infrastructure/postgres/store.go`.
   - Novo método `Engine[S].LoadLatestState(ctx, def, key) (S, Snapshot, bool, error)` — decodifica
     via `Codec[S]` já existente; zero tipo de domínio, zero regra de negócio, zero SQL fora do
     adapter Postgres.
2. **Consumidor (`internal/agents`)**:
   - `workflows.IsResumableAfterFailedWrite(state, now) bool` (novo, `pending_entry_workflow.go`) —
     gate de elegibilidade: `PendingStatusActive`, `Awaiting == AwaitingSlotConfirmation`,
     `len(Candidates) > 0` (categoria já resolvida), não expirado (`isExpired`/`SuspendedAt`, mesmo
     TTL de 30min já usado no fluxo normal — `SuspendedAt` nunca é tocado pelo caminho de escrita,
     confirmado por leitura de todo `executeWrite`/`executeWithIdempotency`/`executeDirectWrite`) e
     `FailedWriteResumeCount < maxFailedWriteResumes` (=2, campo **próprio**, dedicado, para não
     colidir com `ConfirmRepromptCount` — ver Riscos Residuais).
   - `workflows.SeedResumeAfterFailedWrite(state, msg) PendingEntryState` (novo) — seta apenas
     `ResumeText`/`IncomingMessageID`/incrementa `FailedWriteResumeCount`; não toca
     `MessageID`/`ItemSeq`/`OperationKind` (chave de idempotência intacta) nem `Candidates`/
     `CategoryVersion` (sem reclassificação).
   - `PendingEntryContinuer.tryResumeFailedWrite` (novo, `pending_entry_continuer.go`) — quando
     `Resume` retorna `no_run_found`, chama `LoadLatestState`; se `snap.Status == RunStatusFailed` e
     `IsResumableAfterFailedWrite`, semeia o estado com a mensagem recebida e chama `Engine.Start`
     (novo `RunID`, mesma `correlation_key` — permitido porque o índice único parcial
     `workflow_runs_active_key_uidx` só cobre `running`/`suspended`). O `Continue` foi refatorado
     (extração de `resolveResult`) só para caber no limite de 40 statements do linter
     (`function-length`), sem mudança de comportamento nos demais caminhos (marshal-error,
     resume-error, no-run, sucesso).
3. **Reentrada no step existente**: o run revivido reentra em `makePendingEntryStep` com
   `ResumeText != ""` e `Awaiting == AwaitingSlotConfirmation`, roteando para
   `handleConfirmationResume` → `DecideConfirmation` (avalia a mensagem real recebida — aceitar,
   cancelar, reprompt ou expirar) → só em `ConfirmActionAccept` chama `executeWrite`
   (`validateCategoryForWrite` + `executeWithIdempotency`/`executeDirectWrite`, mesmo retry-de-2 já
   testado em 4.0). Nenhuma mensagem arbitrária do usuário dispara escrita sem essa avaliação.

### Por que preserva as 6 restrições do escopo

1. **Sem regressão ADR-002/3.0**: `Engine.Resume` e a propagação de `StepStatusFailed` continuam
   idênticos; erro real de escrita continua indo para `platform_runs.error`/span/log em toda
   tentativa (original e revivida).
2. **Sem swallow-bug**: nenhuma alteração no passo retorna `StepStatusCompleted` mascarando erro.
3. **Sem semântica de domínio no kernel**: `LoadLatest`/`LoadLatestState` operam só sobre
   `Snapshot`/bytes; gates de governança R-WF-KERNEL-001.1-.6 executados e vazios (ver Comandos).
4. **Sem segundo lançamento**: `SeedResumeAfterFailedWrite` preserva
   `MessageID`/`ItemSeq`/`OperationKind`; `IdempotentWrite.Execute` continua fazendo `FindByKey`
   pela mesma chave em qualquer tentativa revivida.
5. **Sem reabrir gate HITL**: a mensagem revivida passa por `DecideConfirmation` antes de qualquer
   escrita — não há bypass de confirmação nem nova pergunta forçada.
6. **Sem reclassificação**: `Candidates`/`CategoryVersion` do snapshot falho são reusados
