## Subtarefas

### 5.1 — Inspeção e correção de `resolveCORSOrigins()`

**Comportamento anterior:** retornava `"*"` quando `CORSAllowedOrigins` estava vazio.

**Problema:** silenciava erro de configuração. Em production, `Config.Validate()` já bloqueia boot se CORS estiver vazio/wildcard (task 1.0), mas o fallback `*` em `resolveCORSOrigins()` criava risco se a validação fosse bypassada.

**Correção:** `resolveCORSOrigins()` agora retorna `cfg.HTTPConfig.CORSAllowedOrigins` diretamente, sem fallback. Se vazio, `httpserver.WithCORS("")` faz `ParseOrigins("")` retornar `[]string{}` — nenhuma origem permitida (seguro). Em production, `Config.Validate()` garante que o valor seja não-vazio e não-wildcard antes do boot.

### 5.2 — Verificação do header `Server:`

Inspecionado `devkit-go@v0.5.0/pkg/http_server/common/security.go`:
- `DefaultSecurityHeaders()` inclui `"Server": ""` no mapa.
- `Apply()` chama `header.Del("Server")` para cada header com valor vazio.
- `securityHeadersMiddleware()` aplica isso a cada resposta.

**Conclusão:** header `Server:` já está suprimido pelo framework. Nenhum middleware adicional necessário.
