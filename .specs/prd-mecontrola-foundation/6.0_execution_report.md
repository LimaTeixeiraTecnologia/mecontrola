## Suposições

- `lazyServerSubsystem` constrói DB e OTel no `Start()` (não no `Bootstrap()`), garantindo que o bootstrap do runtime não falhe em ausência de DB nos testes unitários do runtime.
- O teste de integração usa `RunMigrationsDown` para simular banco "indisponível" (tabela `health_probe` removida), que é a mesma semântica de DB down para o `/ready` handler.
- `stubManager` / `mockManagerHTTP` são stubs inline nos testes (sem mockery) — suficiente para este pacote sem interface exportada complexa.
- `ParseCORSOrigins` filtra wildcard mesmo que esteja no meio da lista (proteção adicional além do devkit-go).

## Riscos Residuais

- O `lazyServerSubsystem.Start()` tenta conectar ao DB e iniciar OTel. Se OTel endpoint estiver configurado como vazio, `NewProvider` retornará erro. Em produção, a variável `OTEL_EXPORTER_OTLP_ENDPOINT` deve estar presente. Em local/test, o endpoint pode ser empty/stub.
- `TestBootstrap/modo_server_deve_inicializar_sem_falha_no_bootstrap` verifica que o `Run` falha quando o banco não está disponível — isso é o comportamento esperado e testado explicitamente.
